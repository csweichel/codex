Understand codex-rs from the command line entrypoint all the way down to tool execution.

This guide is built for readers who do not know Rust. It explains the codebase as a layered system: workspace, crates, binaries, protocols, thread lifecycle, turn execution, tool dispatch, persistence, and the advanced systems that make Codex feel like Codex.

The crucial architectural idea is this: the Rust implementation is not one giant CLI. It is a workspace of many crates, with a modern app-server layer in the middle, multiple clients on top, and a deep core session engine below.

Surface `codex`, `codex exec`, TUI, app-server Core runtime `ThreadManager` -> `CodexThread` -> `Codex` -> `Session` Persistence rollout files + SQLite state Extension model tools, skills, MCP, plugins, multi-agent

If you only remember five things

`codex-rs` is a Cargo workspace, not a single app.

Most folders are separate crates with a narrow responsibility.

The app-server is the central protocol boundary.

TUI and exec increasingly act as clients of a shared RPC/event runtime.

A live conversation is owned by `ThreadManager` and represented by `CodexThread`.

That thread wraps a `Codex` queue-pair and a long-lived `Session`.

`run_turn` is the heart of execution.

It builds context, calls the model, executes tools, streams events, and persists output.

Permissions are explicit data, not hidden side effects.

Thread/turn APIs select permission profiles, approvals, environments, and tool visibility deliberately.

Lesson 1

Workspace shape and the Rust you need to read it

Before looking at behavior, learn how to navigate the repository and how to decode the most important Rust building blocks that appear everywhere else.

How this repository is organized

`codex-rs` is a Cargo workspace. In Rust, a workspace is a single repository that contains many packages and lets them share dependency versions, tooling, and build configuration. The root workspace file is codex-rs/Cargo.toml, and it currently declares 105 member crates.

The naming convention matters: folder names like core/, tui/, and app-server/ become crates named codex-core, codex-tui, and codex-app-server. That is why Cargo dependencies refer to the prefixed names even when the folder omits the prefix.

High-confidence mental model

Think of the workspace as a platform. It is not “the TUI crate plus some helpers.” The platform has protocol crates, client crates, execution/sandbox crates, persistence crates, and extension crates.

Crate clusters to recognize immediately

  • Surfaces: codex-cli, codex-tui, codex-exec, codex-app-server
  • Protocols: codex-protocol, codex-app-server-protocol
  • Core runtime: codex-core, codex-tools, codex-mcp
  • Execution + sandbox: codex-exec-server, codex-sandboxing, codex-shell-command
  • Persistence: codex-rollout, codex-state, codex-thread-store
  • Model/provider layer: codex-model-provider, codex-models-manager

Rust primer: crate, module, binary, library

  • Crate: a compiled package. One folder with one Cargo.toml is usually one crate.
  • Library crate: exposes reusable code from src/lib.rs.
  • Binary crate: executable entrypoint from src/main.rs.
  • Module: a namespace inside a crate, usually a file or folder under src/.
  • `pub`: makes a thing visible outside its module.

Rust primer: the types you will keep seeing

  • `struct`: a named record of fields, like an object with only data.
  • `enum`: a tagged union. One value can be one of several named variants.
  • `impl`: methods attached to a type.
  • `Option<T>`: either Some(value) or None.
  • `Result<T, E>`: either Ok(value) or Err(error).
  • `Arc<T>`: shared ownership of the same object across tasks.
  • `Mutex<T>`: a lock around mutable shared state.

What to read first when opening a new Rust crate

Open Cargo.toml.

It tells you the crate name, whether it is a library or binary, and its direct dependencies.

Open src/main.rs if it exists.

That shows the executable entrypoint and how the crate starts.

Open src/lib.rs.

This is the public face of a library crate and a fast way to see the module map.

Find the main public types and functions.

For codex-rs, these are usually things like run_main, ThreadManager, Session, or a protocol struct.

More depth: why the repo resists adding new code to codex-core Design pressure

The repository instructions explicitly warn against letting codex-core become a dumping ground. That tells you the maintainers think in terms of crate boundaries as architecture, not just package management.

As you read the code, notice the repeated extraction of functionality into crates such as codex-app-server-protocol, codex-tools, codex-exec-server, codex-rollout, and codex-state. That is a deliberate strategy to stop core logic, UI logic, protocol types, and persistence logic from collapsing into a single unreviewable unit.

Lesson 2

Entry surfaces: how the binaries start

The Rust implementation exposes several different user-facing programs. They share a lot of infrastructure, but they are not all the same executable path.

The outermost surface is codex

codex-rs/cli/src/main.rs defines the top-level multitool CLI. It uses Clap, the standard Rust command-line parser, to build a single command that can dispatch to interactive TUI mode, non-interactive exec mode, login commands, MCP commands, app-server, sandbox helpers, update commands, and more.

// High-level shape, simplified MultitoolCli { config_overrides, feature_toggles, remote, interactive, subcommand: Option<Subcommand> }

Important subcommands include Exec, Review, Mcp, McpServer, AppServer, Sandbox, and Resume.

Key startup files

codex-rs/cli/src/main.rs Top-level `codex` parser and command router.
codex-rs/exec/src/lib.rs:221 Non-interactive `run_main`.
codex-rs/tui/src/lib.rs:682 Interactive TUI `run_main`.
codex-rs/app-server/src/lib.rs:353 App-server bootstrap.

Rust primer: what a main() usually does here

  • Parse CLI arguments with Clap.
  • Resolve config overrides.
  • Set up logging/telemetry.
  • Delegate to an async run_main(...) function.
  • Return Result<(), anyhow::Error> so failures can bubble upward cleanly.

What codex-arg0 is doing

Several binaries use arg0_dispatch_or_else from codex-arg0. That means the program can change behavior based on the executable name it was invoked as.

This lets the project ship alias executables or special sandbox helper names without having to duplicate a totally separate codebase for each one.

More depth: why TUI and exec are not “just thin wrappers” Client behavior

They are clients, but they still own substantial surface-specific logic. The TUI owns render state, keyboard handling, overlays, slash-command UX, transcript formatting, and status panes. Exec owns human-output and JSONL-output processors. What they increasingly do not own is the authoritative thread/turn protocol and lifecycle.

That lifecycle is being centralized through the app-server contract so multiple surfaces can rely on the same semantics.

Lesson 3

Vocabulary and protocols: two overlapping languages

codex-rs has both an older internal queue protocol and a newer app-server RPC protocol. They overlap, but they are not identical.

What lives in codex-protocol

codex-rs/protocol/src/protocol.rs defines the internal queue contract used by the core engine: Submission at line 129, Op at line 407, Event at line 1303, and EventMsg at line 1316.

Submission { id, op, trace } Event { id, msg }

Think of this as the native message language of the core runtime. The UI or service submits operations, and the core emits structured events back.

Rust primer: enum-driven protocols

Rust often models protocols with enums rather than stringly typed objects. Instead of “some JSON with a type field,” you get an enum variant like Op::UserInput or EventMsg::TurnComplete.

This catches protocol mistakes at compile time inside Rust code.

What lives in codex-app-server-protocol

The app-server protocol crate defines the external request/response/notification schema. The most important user-driving request types are ThreadStartParams at line 3547 and TurnStartParams at line 5483 of codex-rs/app-server-protocol/src/protocol/v2.rs.

thread/start -> create or re-open a conversation shell turn/start -> submit input to a thread thread/list -> enumerate stored threads thread/read -> read one thread without resuming it turn/interrupt -> cancel an in-flight turn

This contract is JSON-RPC 2.0 shaped: requests, responses, notifications, and server-to-client approval requests.

What JSON-RPC means here

  • Request: client asks the server to do something, like thread/start.
  • Response: server replies once, usually immediately.
  • Notification: server pushes progress updates like turn/completed or item/completed.
  • Server request: the server can also ask the client for approval or user input.

How the vocabularies line up

Concept Internal/core language App-server language Important nuance
Long-lived conversation container Session Thread These are close conceptually, but not literally the same type.
User-facing input cycle Submission of Op::UserInput / Op::UserInputWithTurnContext turn/start The app-server request is translated into a core op.
Progress output EventMsg ServerNotification The app-server maps core events into externally stable notification types.
One internal model/tool loop run_turn Usually exposed as a single app-server turn Externally it looks like one turn, but internally it may include multiple model-tool iterations.
Transcript pieces RolloutItem, ResponseItem, TurnItem Item The app-server simplifies many internal details into the thread/turn/item model.
More depth: a precise runtime handoff Important

In the app-server layer, turn/start is not magic. The handler in codex-rs/app-server/src/codex_message_processor.rs validates the turn overrides, maps app-server UserInput items into core input items, and then submits either Op::UserInput or Op::UserInputWithTurnContext to the target CodexThread.

That means the app-server is not bypassing the core queue model; it is translating into it.

Lesson 3 checkpoint

Which crate defines the app-server v2 request and response types?

Correct answer: codex-app-server-protocol. It defines the typed external RPC schema.

Is the external app-server “turn” always exactly the same as one internal model sampling round-trip?

Correct answer: No. The external turn is the user-facing unit; internally, Codex may loop through multiple model/tool steps before that turn completes.
Lesson 4

The live runtime: `ThreadManager`, `CodexThread`, `Codex`, `Session`

This is the most important ownership chain in the codebase. If you understand this layer, the rest of the runtime stops looking mysterious.

The ownership chain

ThreadManager owns all live threads.

In codex-rs/core/src/thread_manager.rs, the manager keeps a map of ThreadId -> Arc<CodexThread> plus shared services like auth, models, environments, skills, plugins, and MCP.

CodexThread is the thread-scoped façade.

It exposes thread-level actions like submit, next_event, steer_input, and config snapshot access.

Codex is the queue-pair wrapper.

It owns the submission sender, event receiver, agent-status watch channel, and the underlying session.

Session is the long-lived per-thread engine state.

This is where config, mailbox, active turn, realtime conversation, services, hooks, goals, rollout recording, and shared session state live.

Key methods to memorize

core/src/session/session.rs:333 Session::new

What `ThreadManager` centralizes

  • Live thread registry
  • Auth manager
  • Shared models manager
  • Environment manager
  • Skills manager + watcher
  • Plugins manager
  • MCP manager
  • Analytics event client

This makes it the “session runtime factory” for the whole process.

What `SessionConfiguration` carries

  • Model and collaboration mode
  • Approval policy and reviewer
  • Canonical permission profile
  • Working directory and Codex home
  • Dynamic tools
  • Thread name and environments
  • Developer and user instructions
  • Service-tier, personality, compact prompt

The key point is that permissions and context are explicit session data, not spread across random globals.

Rust primer: why you keep seeing Arc<...>

Rust does not let you casually share mutable objects across tasks. When multiple async tasks need the same long-lived object, the code usually wraps it in Arc<T> for shared ownership, then adds a Mutex, RwLock, or channels around the pieces that actually change.

Rust primer: channels and watches

  • Submission sender: sends commands into the session loop.
  • Event receiver: yields structured events back out.
  • watch::Sender/Receiver: tracks the latest status, such as agent state, without replaying the whole history.
More depth: `Session::new` does real initialization work Startup details

Session::new is not a trivial constructor. It performs session persistence setup, state DB lookup, history metadata loading, auth lookup, and MCP server resolution in parallel using tokio::join!.

It also builds session trace metadata, loads rules/skills/plugins, resolves the effective model, derives base instructions, materializes dynamic tools, and then starts the background submission loop that processes future ops.

Lesson 4 checkpoint

Which component actually owns the map of live threads?

Correct answer: ThreadManager. It is the process-level owner of live thread objects and the shared services they depend on.

What is the best description of Codex in this architecture?

Correct answer: a queue-pair wrapper around submissions, events, and a Session. It is the direct conduit between callers and the session loop.
Lesson 5

Turn execution: what happens after you submit a prompt

This is the “agent loop” most people are trying to find when they ask how Codex works. In codex-rs, the key implementation lives in core/src/session/turn.rs.

The end-to-end sequence

A client submits user input.

For app-server clients, this begins as turn/start. Internally it becomes Op::UserInput or Op::UserInputWithTurnContext.

The session handler creates or updates turn context.

user_input_or_turn_inner validates overrides, creates a turn context, and either steers an active turn or spawns a new regular task.

run_turn prepares context.

It may compact history, record context updates, inspect plugins and connectors, resolve skills, and compute turn-scoped instructions.

The model is called.

The session creates or reuses a model client session, sends the prompt/context/tools, and starts consuming streaming response events.

Tool calls are parsed and executed.

Function calls, MCP calls, local shell calls, and custom tools are routed through the tool system and their outputs are fed back into the loop.

Streaming deltas are emitted to clients.

Assistant text, reasoning text, tool progress, and completion items become events and notifications.

Final output is recorded and persisted.

Turn items are written to rollout storage, status is updated, token usage is tracked, and the turn completes.

Key files

core/src/session/handlers.rs Turns app-server/core ops into session actions.
core/src/session/turn.rs:136 run_turn
core/src/stream_events_utils.rs Maps model output into recorded items and deltas.
core/src/tools/router.rs Chooses what a tool call means.

Three jobs of run_turn

  • Build model-visible context: instructions, skills, plugins, permissions, prior history, dynamic tools.
  • Run the model/tool loop: stream response items, execute tool calls, and continue when outputs create more work.
  • Emit durable events: persist rollout items and deliver real-time updates to clients.

Rust primer: async and cancellation

run_turn is an async fn, which means it returns a future that runs on Tokio’s async runtime. Cancellation is explicit: many operations receive a CancellationToken so interrupts can stop a turn cleanly.

One subtle but crucial point

A single turn is not “call model once, print answer.” The model may respond with tool calls, those tools may produce outputs, and those outputs may require another model step. The loop continues until the model produces a final assistant message and there is no more pending work to feed back in.

Why this matters for reading bugs

Many “why did the agent do X?” questions are really questions about a multi-stage turn. You need to inspect the item stream, not only the final assistant text.

More depth: where app-server input becomes a core turn Bridging detail

In codex-rs/app-server/src/codex_message_processor.rs, the turn_start handler loads the target thread, validates overrides, converts v2 input items to core input items, and chooses between Op::UserInput and Op::UserInputWithTurnContext.

That op is then submitted to the target CodexThread via submit_with_trace. The core session loop does the rest.

Lesson 5 checkpoint

What is the best description of run_turn?

Correct answer: it is the core model/tool/event loop for a user turn.

Can one user-visible turn include multiple internal model/tool iterations?

Correct answer: Yes. Tool calls and their outputs can lead to additional model steps before the turn ends.
Lesson 6

Tools, MCP, sandboxes, and approvals

This is the action layer: what the model is allowed to call, how those calls are interpreted, and how Codex decides whether real side effects should proceed.

Tool specs are model-visible contracts

In codex-rs/tools/src/tool_spec.rs, the workspace defines model-visible tool schemas like:

  • Function
  • Namespace
  • ToolSearch
  • LocalShell
  • ImageGeneration
  • WebSearch
  • Freeform custom tools

These are what the model “sees” as callable capabilities.

ToolRouter turns specs into behavior

core/src/tools/router.rs builds the actual turn-local router from the current config, MCP tools, deferred tools, discoverable tools, and dynamic tools.

When a model response item contains a tool call, ToolRouter::build_tool_call decides whether it is:

  • a local function tool,
  • an MCP tool call,
  • a local shell execution,
  • a tool-search request,
  • or a freeform custom tool call.

MCP in the architecture

  • McpManager resolves configured/effective servers from config and plugins.
  • codex-mcp owns connection/runtime mechanics.
  • Sessions can expose MCP tools to the model and call them during turns.
  • The app-server also exposes direct MCP resource and tool RPCs.

Permissions are data structures

  • Legacy fields: sandbox / sandboxPolicy
  • Canonical field: PermissionProfile
  • Thread and turn APIs can select or modify permission profiles
  • Session configuration stores the active permission profile and reviewer

Approvals have multiple layers

  • User-facing approval prompts
  • request_user_input tool questions
  • Permission adjustment requests
  • Guardian automatic approval review for certain on-request actions

Guardian review is a separate review system, not a boolean check

The guardian subsystem reconstructs a compact transcript, spawns or reuses a dedicated review session, asks for strict JSON output, and fails closed if anything goes wrong.

That means auto-approval is itself implemented as an agentic review path with its own prompt, model preference, timeout, transcript cursor, and circuit breaker for repeated denials.

Rust primer: tagged unions are why tool calls stay readable

When you see a Rust enum like ToolSpec or ToolPayload, each variant can carry different data. That is how the code cleanly represents “this tool call is a local shell call” versus “this tool call is an MCP call.”

Rust primer: `serde` keeps protocol types honest

serde is the Rust serialization framework used across this repo. It turns Rust structs and enums into JSON and back again, which is why protocol crates can share exact request/response shapes with clients.

More depth: parallel tools and deferred tools Advanced

The tool router distinguishes between tools that support parallel execution and tools that do not. For MCP tools, parallel capability is configured per server. For local configured tools, parallel support is attached to the tool spec.

Deferred dynamic tools are also filtered from the model-visible set until they are intentionally loaded, which lets the system advertise a controlled capability surface instead of eagerly exposing everything.

Lesson 6 checkpoint

Which component decides how a model-emitted tool call should be interpreted and routed?

Correct answer: ToolRouter. It builds turn-local tool behavior from specs and runtime context.

Is guardian auto-approval review just a simple if-statement on the current command?

Correct answer: No. Guardian review is implemented as its own review session and policy prompt flow.
Lesson 7

App-server and clients: the shared middle layer

The modern architectural center of codex-rs is not the TUI. It is the app-server contract and the clients that sit on top of it.

What the app-server bootstraps

In codex-rs/app-server/src/lib.rs, run_main_with_transport_options creates the environment manager, transport channels, outgoing router, config manager, feedback/telemetry pipeline, and the message processor stack.

It explicitly uses two loops/tasks: a processor loop for incoming JSON-RPC and an outbound loop for writing messages to connections.

Message processing is layered

  • MessageProcessor: connection/session gating, initialize flow, config, FS, device keys, external agent config, and RPC orchestration.
  • CodexMessageProcessor: thread/turn/app/model/plugin/MCP behavior specific to Codex.
  • ThreadManager: actual live thread ownership below both processors.

Why the TUI talks to app-server even in process

codex-app-server-client provides an in-process client facade. It does not bypass app-server semantics; it preserves the same request/response/event model while removing the process boundary.

That lets TUI and exec share the same protocol behavior while still enjoying cheap in-process calls when appropriate.

Important design choice

The in-process path still uses JSON-RPC result envelopes internally on purpose. The goal is to preserve one contract, not invent a second private one.

Key client methods

TUI-specific client logic

The TUI is built around three main layers:

  • App: top-level runtime loop and orchestration.
  • AppServerSession: typed request wrapper for the app-server.
  • ChatWidget: transcript rendering and event-driven history state.

The TUI keeps a canonical internal ThreadSessionState derived from app-server responses and notifications.

Exec-specific client logic

Exec mode starts the same app-server/client machinery but swaps in event processors that render to either:

  • human-readable stderr/stdout output, or
  • JSONL event output for automation.

That is why exec mode still gets the full thread/turn/item lifecycle without needing TUI state.

Backpressure handling is a real architectural concern

The in-process client explicitly classifies some events as lossless and others as best-effort. Transcript deltas and completion notifications must be delivered even when the consumer is slow; some progress events may be dropped with a lag marker instead.

This is an unusually strong sign of maturity in the architecture: the authors have already encountered queue saturation and designed around it.

More depth: exact thread/turn request translation Precise path

thread/start in CodexMessageProcessor loads config overrides, selects environments, calls ThreadManager::start_thread_with_options, snapshots the created thread, attaches a listener, and sends both a response and a thread/started notification.

turn/start loads the thread, validates overrides, maps input, submits a core op to the thread, records the resulting submission id as the turn id, and responds immediately with an in-progress turn shell.

Lesson 7 checkpoint

What is the best description of codex-app-server-client?

Correct answer: a shared client facade that can speak to app-server in process or remotely.

Why does the in-process client still preserve app-server semantics instead of inventing a private shortcut API?

Correct answer: one behavioral contract across surfaces. The in-process client removes the process boundary, not the protocol semantics.
Lesson 8

Persistence and advanced systems

This is where the codebase stops being “a chat loop” and becomes an operating environment: rollouts, SQLite state, rollout traces, multi-agent, goals, and context assembly.

Rollout persistence

The codex-rollout crate handles session-file persistence and discovery. It exposes concepts like session directories, archived session directories, cursors, thread listing, and the RolloutRecorder.

This is the durable “what happened in this thread?” layer.

SQLite state

The codex-state crate mirrors rollout metadata into SQLite-backed state. The crate docs explicitly say it is intentionally small and focused: rollout scanning/backfill lives in core, while the state crate owns the storage engine and runtime access.

This is how Codex tracks metadata like thread info, dynamic tools, goals, spawn edges, device keys, and remote-control enrollment.

Rollout trace

Rollout trace is an opt-in local diagnostic system. It records raw runtime evidence into a bundle and later reduces it into a semantic graph.

The key design phrase from the docs is: observe first, interpret later.

Multi-agent is rooted, not global

AgentControl is shared across a root thread tree. It is the control-plane handle for spawning agents, delivering messages, tracking metadata, and routing completion envelopes back to parents.

This is important: the registry is scoped to a root session tree, not the whole process.

Context assembly is modular

The context/ and context_manager/ modules show how Codex builds model-visible instructions from many fragments:

  • permissions instructions
  • apps and plugins instructions
  • skills instructions
  • personality and collaboration-mode instructions
  • turn-aborted markers and subagent notices
  • hook-provided additional context

Rust primer: why small crates matter here

In a language with strong compile-time boundaries, crate design is part of system design. Splitting rollout, state, tools, protocol, and app-server into dedicated crates makes it easier to enforce architecture through the type system and dependency graph.

Rust primer: “best effort” and fail-closed behavior are explicit

You will often see the code spell out its safety posture. For example, rollout tracing is best effort and must never break a session; guardian review fails closed; config overrides are validated before a turn is accepted. This explicitness is a recurring Rust engineering style in the repo.
